description: Record a backlog of improvements, bugs, and ideas entered one at a time, written as Markdown or HTML, or extend an existing backlog.
argument-hint: "[existing-backlog-file]"
---

Record a project backlog — improvements, bugs, and ideas captured for later planning — from items entered one at a time, written to a Markdown or HTML file. Follow these steps exactly and do not skip the interactive prompts.

## 1. Determine whether to create a new backlog or extend an existing one

- If a filename was provided as an argument (`$ARGUMENTS`), treat it as an existing backlog to extend:
  - If the file does not exist, stop and say so.
  - Read it fully to understand its title, structure, conventions, and existing items.
  - Detect its format from the extension: `.md` → Markdown, `.html` → HTML.
  - **Skip** the title question (step 2) and the output format question (step 5): reuse the existing title and write back to the same file in its existing format. The new items are appended to the existing backlog (see step 6).
- If no filename was provided, ask the user with `AskUserQuestion` as a **single-select** question whether to **create a new backlog** or **extend an existing one**:
  - **Create a new backlog** — continue with the title question below.
  - **Extend an existing backlog** — find candidate backlog files in the repository and ask the user which one to use, then treat the chosen file as an existing backlog to extend exactly as in the argument case above (read it fully, detect its format, skip steps 2 and 5). Finding candidates needs a Git repository — if the current directory is not inside a Git working tree (`git rev-parse --is-inside-work-tree`), tell the user to pass the backlog file as an argument instead.
    - List `*.md` / `*.html` files via `git ls-files '*.md' '*.html' ':(exclude)Docs/agent-plans/completed/**'` plus untracked ones via `git ls-files -o --exclude-standard '*.md' '*.html' ':(exclude)Docs/agent-plans/completed/**'`, then keep only those that actually look like backlogs — inspect each candidate's contents rather than guessing from the name. Exclude standard repository boilerplate (README, LICENSE / COPYING, CODE_OF_CONDUCT, CONTRIBUTING, CHANGELOG, SECURITY, and similar conventional project files, regardless of extension or directory), plan files (names ending in `-plan.md` / `-plan.html`), source files that already have an associated plan (e.g. exclude `backlog.md` when `backlog-plan.md` or `backlog-plan.html` exists alongside it), and files generated by the other commands (`code-review*`, `changes*`, and `release-notes*`).
    - If a `Docs/agent-plans/active` directory exists at the repository root, favor the candidates found under it: list those first and mark them as recommended, but keep listing the candidates found elsewhere too. (Finished initiatives under `Docs/agent-plans/completed` are already excluded by the `:(exclude)` pathspec above, so they never reach this list.)
    - Present them with `AskUserQuestion` as a **single-select** question. If no candidates exist, tell the user there is no existing backlog to extend and offer to create a new one instead.

## 2. Choose the backlog title

- Only when creating a new backlog (when extending an existing one, reuse its title).
- Ask the user for the backlog title. Use this title for the document and to derive the backlog path in step 5.

## 3. Collect the backlog items

- Collect the items one at a time, prompting the user to type each item as free text. The user signals the end by typing `finish` or `done` as an item.
- After capturing an item, repeat the prompt for the next one. Continue until the user signals the end.
- Require at least one item. If the user signals the end before entering any item, ask them to add at least one.
- Capture each item's text verbatim as the user enters it; do not rewrite it yet. You will refine wording in step 4.

## 4. Understand the items

- If the current directory is inside a Git repository (`git rev-parse --is-inside-work-tree`), take a quick look at the repository to understand the overall scope and purpose of the project. Keep this fast — get a sense of what the project is (e.g. README, top-level structure), and do not dive deep into the source code.
- Ask the user any clarifying questions needed to capture each item accurately. Use the answers only to refine the items — do not record the questions or answers in the backlog.
- When extending an existing backlog, also read its existing items so the new ones are consistent with what is already there and not redundant.
- If you are not inside a repository, skip the repository analysis but still ask clarifying questions if an item is ambiguous.

## 5. Choose the output format and derive the backlog path

- Only when creating a new backlog (when extending, reuse the existing file and format from step 1).
- Use `AskUserQuestion` as a **single-select** question to ask whether the backlog should be written in **Markdown** or **HTML**.
- **Choose the location.** When inside a Git repository, determine the repository root (`git rev-parse --show-toplevel`) and use `AskUserQuestion` as a **single-select** question to ask where to write the backlog, offering exactly these two options (always include both, whether or not `Docs/agent-plans/active` already exists):
  - **Repository root** — write directly to the repository root.
  - **Docs/agent-plans** — write into a dated subfolder of `Docs/agent-plans/active`, creating the `Docs/agent-plans/active` directory (and the subfolder) if it does not already exist.
  - When not inside a Git repository, do not ask: use the current directory.
  - This location question and the Markdown/HTML question above may be asked together in a single `AskUserQuestion` call.
- **Derive the slug** from the title: lowercase it, replace any run of non-alphanumeric characters with a single hyphen, and trim leading/trailing hyphens. Use the extension `md` or `html` per the chosen format.
- **Derive the path** from the chosen location:
  - **Repository root / current directory:** write `<slug>.<ext>` directly in that directory.
    - title `Project Phoenix`, Markdown → `project-phoenix.md`
    - title `Q3 2026 Backlog`, HTML → `q3-2026-backlog.html`
  - **Docs/agent-plans:** create a subfolder named `<YYYY-MM-DD>-<slug>` (using today's date) under `Docs/agent-plans/active` and write the file as `<slug>.<ext>` inside it.
    - title `Some Feature`, Markdown, on 2026-05-17 → `Docs/agent-plans/active/2026-05-17-some-feature/some-feature.md`
- Never overwrite an existing file. Append the smallest integer suffix that does not collide, applied to whichever part of the path keeps backlogs from clashing:
  - **Repository root / current directory:** suffix the filename (`project-phoenix-2.md`, `project-phoenix-3.md`, …).
  - **Docs/agent-plans:** suffix the dated subfolder, keeping the filename inside it unchanged (`Docs/agent-plans/active/2026-05-17-some-feature-2/some-feature.md`, …).
  - Report the final path used.

## 6. Write the backlog

- When creating a new backlog, structure it using the template in the next step, rendering it idiomatically in the chosen output format, and write it to the derived path.
- When extending an existing backlog, append the new items to the existing document, matching its existing structure and formatting rather than imposing the template. Keep all existing items unchanged, add the new ones in the same style, and update the document's generated/updated date if it has one.

## 7. Backlog template

Every backlog has the same structure regardless of the output format chosen in step 5. The template below defines the **content** the backlog must contain — not its formatting. Render each item idiomatically in the chosen format (Markdown or HTML), and replace every `<...>` placeholder.

Keep the backlog focused on the items themselves. Do not add technical or implementation details on your own — include them only when they are absolutely relevant or explicitly provided by the user.

- **Title** — `<backlog title>`
- **Generated** — `<YYYY-MM-DD>`
- **Overview** — an optional paragraph or two describing the purpose and theme of the backlog. Include only if it adds value; omit otherwise.
- **Backlog** — the core of the document: the items in order, repeated one entry per item. Each item contains:
  - **Title** — `<short title for the item>`
  - **Description** — what the item is and why it matters, refined from the user's text with the context gathered in step 4.

### HTML rendering

When the chosen output format is HTML, render the backlog as a self-contained document with an inline `<style>` block (no external assets), using a dark theme: a dark background with light, comfortably readable text.

## 8. Open the backlog file

- After writing the file, open it in the platform's default application so it appears for the user right away. Pick the command that matches the current platform: `open "<path>"` on macOS, `xdg-open "<path>"` on Linux, or `start "" "<path>"` on Windows. Use the final path derived in step 5, including any collision suffix. When extending an existing backlog, open that same file.

## 9. Summarize for the user

- After writing the file, present the newly created items inline in the conversation so the user can review them at a glance — title and a short description for each. Summarize where necessary to keep it concise.
- When extending an existing backlog, show only the items that were just added, not the pre-existing ones.
