---
description: Implement a plan file milestone by milestone, pausing after each for manual review and commit.
argument-hint: "[plan-file]"
---

Implement a milestone-based plan, one milestone at a time. The plan can be any Markdown or HTML plan file. Follow these steps exactly.

## 1. Determine the plan file

- If a filename was provided as an argument (`$ARGUMENTS`), use it as the plan file.
- Otherwise, find candidate plan documents in the repository and ask the user which one to use. This fallback needs a Git repository — if the current directory is not inside a Git working tree (`git rev-parse --is-inside-work-tree`), stop and ask the user to pass a plan file as an argument instead:
  - List plan files (names ending in `-plan.md` / `-plan.html`) via `git ls-files '*-plan.md' '*-plan.html' ':(exclude)Docs/agent-plans/completed/**'` plus untracked ones via `git ls-files -o --exclude-standard '*-plan.md' '*-plan.html' ':(exclude)Docs/agent-plans/completed/**'`. If none are found, fall back to listing all `*.md` / `*.html` files (keeping the `:(exclude)Docs/agent-plans/completed/**` pathspec on both commands), excluding standard repository boilerplate (README, LICENSE / COPYING, CODE_OF_CONDUCT, CONTRIBUTING, CHANGELOG, SECURITY, and similar conventional project files, regardless of extension or directory) and files generated by the other commands (`code-review*`, `changes*`, and `release-notes*`), which are not plans.
  - Exclude plans that are already fully completed — those where every milestone's **Status** is `Completed` — since there is nothing left to implement. Inspect each candidate's contents to decide. If this leaves no candidates (every plan is complete), say so rather than presenting a completed plan.
  - If a `Docs/agent-plans/active` directory exists at the repository root, favor the candidates found under it: list those first and mark them as recommended, but keep listing the candidates found elsewhere too. (Finished initiatives under `Docs/agent-plans/completed` are already excluded by the `:(exclude)` pathspec above, so they never reach this list.)
  - Present them with `AskUserQuestion` as a **single-select** question. If no candidates exist, stop and tell the user.
- If the chosen file does not exist, stop and say so.

## 2. Choose the implementation approach

- Ask the user with `AskUserQuestion` whether to implement using TDD (test-driven development) or not. Offer both options clearly:
  - **TDD** — write the tests first, watch them fail, then implement until they pass.
  - **Without TDD** — implement directly, adding tests as appropriate.
- Use the chosen approach for every milestone in the next step.

## 3. Implement

Read the plan file, then implement the milestones one by one using the approach chosen in step 2. For each milestone, follow this cycle exactly:

1. **Review past feedback** — before implementing, always re-read the **Review comments**, **Automated review**, and **Implementation notes** of the milestones already marked `Completed` in the plan file. Never skip this: the user's own review comments are the most important input here, so read every one of them. Extract any guidance that generalizes beyond the milestone it was recorded in — recurring corrections, conventions the user enforced, decisions about structure or style — and apply it to this milestone so you do not reproduce issues that were already addressed. Skip feedback that was purely specific to a past milestone and does not apply here. For the first milestone, there is nothing to review yet.
2. **Implement** — carry out the milestone's tasks using the chosen approach.
3. **Automated review loop** — before recording progress or pausing for the user, run an automated review of the milestone's work and address what it finds, iterating until it converges. This loop is **fully automated**: do not involve the user in it. Repeat the following cycle up to a maximum of **3 iterations**:
   - **Dispatch a fresh-context reviewer.** Spawn a sub-agent (via the Agent tool) with **no prior conversation context**, so it reads the code cold as an independent reviewer would. Provide it with:
     - **Scope** — review only this milestone's uncommitted changes: the union of the staged changes (`git diff --cached`), the unstaged changes to tracked files (`git diff`), and the untracked files (`git ls-files -o --exclude-standard`). Untracked files do not appear in `git diff`, so the reviewer must read them directly. Include only the scopes that actually contain files.
     - **Context** — this milestone's tasks and goals from the plan, plus the generalizable feedback gathered in step 1, so the reviewer understands the intent and the conventions already enforced.
     - **Instructions** — perform a complete, in-depth review of those changes. Read the relevant source code and never assume. Consult the Git history when a decision's rationale is unclear. Back every finding with evidence from the code. Cover architecture, API, performance, security, stability, crashes, undefined behavior, and style. Explicitly check for **regressions**: behavior, APIs, or guarantees that previously worked and are now altered, removed, or broken, including impacts on existing callers and tests.
     - **Report format** — return the findings directly in its response; it must **not** write any file. For each finding: a short title, a **Severity** (`Critical`, `High`, `Medium`, `Low`, or `Info`), a category, the location (`path/to/file:line`), a description of what the issue is and why it matters, and a concrete recommendation. If nothing is wrong within the scope, it must say so rather than invent findings.
   - **Act on the findings.** Fix every finding of severity **Critical, High, or Medium**, using the same conventions and the chosen implementation approach (TDD or not). Findings of severity **Low** or **Info** do not require a fix — keep note of them so they can be surfaced to the user at the pause.
   - **Decide whether to continue.** If the latest review reported no `Critical`/`High`/`Medium` findings, the loop has converged — stop. Otherwise, if fewer than 3 iterations have run, repeat the cycle with a **new** fresh-context sub-agent reviewing the updated state; each iteration must use a new sub-agent so the re-review is genuinely independent. After 3 iterations, stop even if `Critical`/`High`/`Medium` findings remain, and carry the unresolved ones into the pause for the user to decide.
4. **Record progress** — update that milestone in the plan file in place:
   - Set its **Status**, and fill in its **Implementation notes** with what was done, the decisions made, and any iterations. Include a brief summary of the automated review loop: how many rounds ran and the overall outcome.
   - Whenever you change a milestone's **Status**, update its entry in the plan's **Table of contents** to the same status, so the index never drifts out of sync with the milestones. If the plan predates this convention and has no table of contents, skip this — do not add one.
   - Record the automated review's findings in that milestone's **Automated review** section — kept separate from the **Review comments** section, which holds human feedback recorded in step 6. For each finding, capture its title, severity, and what it flagged, and note its **resolution status**: `Fixed` for findings that were addressed, `Deferred` for `Low`/`Info` items left for the user to weigh in on, and `Unresolved` for any `Critical`/`High`/`Medium` finding still open after the iteration cap. If the section already has entries, append to it rather than overwriting.
5. **Pause for review** — stop and let the user review the work. Surface the `Low`/`Info` findings and any `Critical`/`High`/`Medium` findings still unresolved after the automated loop's iteration cap, so the user can weigh in. Do not move on to the next milestone yet.
6. **Handle review feedback** — whenever the user gives feedback on the milestone (a change request, a correction, a comment, or a question about the work):
   - **Record it first**, before you reply to it or act on it. Rewrite the feedback in clear, concise terms — preserving its intent and meaning — and append it to that milestone's **Review comments** section in the plan file (the human-feedback section, distinct from the **Automated review** section filled in step 4). Never overwrite existing entries. Do this before you answer, push back, or start any change — no exceptions.
   - Only then respond: apply the requested changes and update the milestone's **Implementation notes** with what changed. If the feedback was a genuine question rather than a change request, answer it and make no code changes until the user confirms one is wanted.
   - Pause for review again, and repeat this step until the user is satisfied with the milestone.
7. **Complete** — once the user is satisfied, set the milestone's **Status** to `Completed`, update its **Table of contents** entry to `Completed` to match, and record its **Completed** date. Then provide the commit message (below) and stop. Do not start the next milestone yet: wait for the user to review, commit manually, and explicitly tell you to continue. Only move on to the next milestone once they have confirmed.

Continue milestone by milestone until the plan is complete. Once the last milestone is `Completed` and every milestone in the plan is now `Completed`, proceed to step 4 to offer archiving the plan.

### Commit message

When a milestone is complete, provide a commit message for it:

- Follow Conventional Commits. Look at the previous commits (`git log`) for the types, scopes, and style actually used in this repository, and match them.
- Use a concise, imperative subject line, followed by a body that describes the changes. Always include a description in the body, explaining what changed and why. Do not mention milestones in the commit message.
- Add a co-author trailer in the standard git form `Co-Authored-By: Name <email>`, on its own line after a blank line at the end of the message, using your own model name and the Anthropic no-reply address `noreply@anthropic.com`.
- Output the proposed commit message inside a fenced code block so it is easy to copy. Every line of the message must start at the left margin with no indentation or leading whitespace, so it can be copied and pasted verbatim.
- Do not stage, commit, or push anything yourself — the user reviews and commits each milestone manually.

## 4. Archive the completed plan

Once every milestone in the plan is `Completed`, the initiative is finished and its files can move out of the active area so future runs of these commands no longer have to consider it.

- This step applies **only** when the plan file lives inside a dated subfolder of `Docs/agent-plans/active` (e.g. `Docs/agent-plans/active/2026-05-17-some-feature/`). If the plan is anywhere else — the repository root, or any directory outside `Docs/agent-plans/active` — there is nothing to archive: skip this step entirely.
- Use `AskUserQuestion` as a **single-select** question to offer moving the finished initiative to `Docs/agent-plans/completed`. Make clear it is optional, and that declining simply leaves the files where they are.
- If the user declines, stop — leave everything in place.
- If the user accepts, move the **entire dated subfolder** (with the source document, the plan, and anything else it contains) from `Docs/agent-plans/active` to `Docs/agent-plans/completed`, keeping the subfolder's name unchanged:
  - Create `Docs/agent-plans/completed` if it does not already exist.
  - `Docs/agent-plans/active/2026-05-17-some-feature/` → `Docs/agent-plans/completed/2026-05-17-some-feature/`.
  - If a subfolder of the same name already exists under `Docs/agent-plans/completed`, append the smallest integer suffix that does not collide to the moved subfolder (`2026-05-17-some-feature-2`, …), and report the final path used.
  - Use `git mv` for tracked files so history is preserved, falling back to a plain move for untracked ones.
- Do not stage, commit, or push the move — the user reviews and commits it manually, just like each milestone.
